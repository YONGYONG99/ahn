# 03. HTTP 메서드

## HTTP 메서드란?

클라이언트가 서버에게 **"뭘 하고 싶은지"** 알려주는 동사입니다.

---

## 핵심 4가지 메서드

| 메서드 | 의미 | DB 동작 | 예시 |
|---|---|---|---|
| `GET` | 조회 | SELECT | 게시글 목록 불러오기 |
| `POST` | 생성 | INSERT | 게시글 작성하기 |
| `PUT` | 전체 수정 | UPDATE | 게시글 전체 수정 |
| `PATCH` | 일부 수정 | UPDATE | 제목만 수정 |
| `DELETE` | 삭제 | DELETE | 게시글 삭제하기 |

---

## 각 메서드 상세

### GET — 데이터 조회

```
GET /posts        → 게시글 전체 목록
GET /posts/1      → 1번 게시글
```

- 바디(Body) 없음
- 서버의 데이터를 **변경하지 않음**
- 브라우저 주소창에 URL 입력하는 것도 GET 요청

---

### POST — 데이터 생성

```
POST /posts
Body: {
  "title": "첫 번째 글",
  "content": "안녕하세요"
}
```

- 바디에 생성할 데이터를 담아 보냄
- 서버에서 새 데이터가 **생성**됨
- 호출할 때마다 새로운 데이터가 만들어짐

---

### PUT — 데이터 전체 수정

```
PUT /posts/1
Body: {
  "title": "수정된 제목",
  "content": "수정된 내용"
}
```

- 해당 자원의 **전체를 교체**
- 빠진 필드는 null 또는 기본값으로 처리될 수 있음

---

### PATCH — 데이터 일부 수정

```
PATCH /posts/1
Body: {
  "title": "제목만 바꿀게요"
}
```

- 보낸 필드만 수정, 나머지는 그대로 유지
- PUT보다 실용적인 경우가 많음

---

### DELETE — 데이터 삭제

```
DELETE /posts/1
```

- 바디 없이 URL만으로 삭제 대상을 지정
- 보통 성공 시 `204 No Content` 응답

---

## 멱등성 (Idempotent)

같은 요청을 여러 번 보내도 결과가 같은 성질입니다.

| 메서드 | 멱등성 |
|---|---|
| `GET` | ✅ (몇 번 조회해도 데이터 변화 없음) |
| `PUT` | ✅ (같은 데이터로 수정하면 결과 동일) |
| `DELETE` | ✅ (이미 삭제된 것을 다시 삭제 요청해도 결과 같음) |
| `POST` | ❌ (호출마다 새 데이터 생성됨) |
| `PATCH` | ❌ (경우에 따라 다름) |

> 실무에서 멱등성을 고려하면 API 설계가 안정적이 됩니다.

---

## 실전 예시 — 블로그 API 설계

```
# 게시글 CRUD
GET    /posts          → 목록 조회
GET    /posts/1        → 단건 조회
POST   /posts          → 게시글 작성
PUT    /posts/1        → 게시글 전체 수정
PATCH  /posts/1        → 게시글 일부 수정
DELETE /posts/1        → 게시글 삭제

# 댓글 CRUD (중첩 자원)
GET    /posts/1/comments     → 1번 게시글 댓글 목록
POST   /posts/1/comments     → 댓글 작성
DELETE /posts/1/comments/3   → 댓글 삭제
```

---

## 정리

```
자원(URL)  +  행동(메서드)  =  API 엔드포인트

GET  /users/1   → 1번 유저 조회
POST /users     → 유저 생성
DELETE /users/1 → 1번 유저 삭제
```
